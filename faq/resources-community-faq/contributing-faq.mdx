# How can I contribute or report issues?

Contributing to GoogleTest and GoogleMock is a valuable way to help improve the framework, fix bugs, add features, and enhance documentation. This guide provides clear instructions on how to effectively contribute, report issues, and adhere to community and project standards.

---

## Contributing Code and Documentation

### Getting Started

Before contributing code or documentation, ensure you have the latest version of the repository and are familiar with the project's [README](README.md) and general guidelines.

- Fork the repository and clone your fork.
- Create a feature branch to work on your changes.
- Make your changes adhering to the project's coding standards and style.
- Test your changes thoroughly.
- Commit your changes with descriptive messages.

### Coding Standards and Style

GoogleTest follows strict C++ style guidelines consistent with Google's C++ style.

- Follow naming conventions and design patterns used throughout the project.
- Write clear, maintainable, and well-documented code.
- Ensure all public interfaces have descriptive comments.
- Avoid introducing unnecessary dependencies.

### Documentation Contributions

Good documentation is crucial. You can contribute by:

- Fixing typos and grammatical errors.
- Adding clarifications, examples, or explanations.
- Extending existing pages to cover new features or common questions.
- Keeping documentation consistent with the latest codebase.

Please follow Markdown formatting standards used in existing documents.

### Testing Your Changes

- Run the full test suite before submission.
- Add tests if you introduce new features or fix bugs.
- Verify that your changes do not break existing tests.

## Reporting Bugs and Issues

### Before Filing an Issue

- Search existing issues and pull requests to avoid duplicates.
- Check the [FAQ](faq.md) and [Getting Started](getting-started/core-setup/prerequisites-requirements.md) guides for known problems.

### Writing a Good Issue Report

- Provide a clear and descriptive title.
- Explain the problem concisely and clearly.
- Include steps to reproduce the issue.
- Provide relevant code snippets or configurations.
- Include platform, compiler, and environment details.
- Attach test failures, error messages, or logs if applicable.

### Feature Requests

When proposing new features:

- Describe the use case and benefits.
- Explain alternatives considered.
- Avoid vague requests; be as concrete as possible.

## Submitting Pull Requests (PRs)

- Ensure your PR includes tests and documentation updates if relevant.
- Follow commit message guidelines.
- Describe changes and their rationale clearly in the PR description.
- Link to any related issues.

## Code Review and Approval

All contributions undergo peer review. Be prepared to:

- Respond to code review comments.
- Update your PR based on feedback.
- Engage constructively with reviewers.

## Community Code of Conduct

Contributions and interactions should follow the project's code of conduct, emphasizing respect, professionalism, and collaboration.

## Summary

Contributing to GoogleTest involves following coding and documentation standards, thorough testing, and engaging with the community constructively. Reporting bugs or feature requests requires detailed, clear descriptions, reproducible examples, and contextual information. Pull requests should be well-scoped and accompanied by tests and documentation where applicable.

## Community and Support

For questions, discussions, and unofficial support, the community resources include:

- GitHub Issues and Pull Requests for reporting and discussion.
- Mailing lists and forums linked from the official GitHub repository.
- Community-created documentation and tutorials.

Engaging with these resources will not only clarify contribution pathways but also help you stay updated with ongoing developments.

## Best Practices for Contributors

- Keep your branches up to date with the main branch.
- Write clear, explicit commit messages.
- Keep pull requests small and focused.
- Run all tests before submitting.
- Respect code ownership and maintainers' feedback.
