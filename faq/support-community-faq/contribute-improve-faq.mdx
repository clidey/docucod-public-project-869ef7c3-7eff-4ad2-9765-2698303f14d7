# How to Contribute to GoogleTest/GoogleMock

This guide summarizes the process for contributing code, documentation, or reporting issues to the GoogleTest and GoogleMock projects. Successful contributions adhere to project guidelines, improve functionality or documentation, and maintain the quality and reliability of the testing framework.

---

## Overview of Contribution Process

Contributing to GoogleTest/GoogleMock involves several key steps intended to ensure smooth collaboration:

1. **Familiarize Yourself with the Project**: Understand the goals, architecture, and coding standards of the project. Reviewing the existing documentation and code base helps align your contributions with project expectations.

2. **Choose Your Contribution Type**:
   - **Code contributions**: Bug fixes, feature enhancements, or performance improvements.
   - **Documentation contributions**: Clarifications, new guides, FAQs, or API references.
   - **Reporting issues**: Bug reports, feature requests, or suggesting improvements.

3. **Follow the Contributing Guidelines**: The project maintains a detailed [CONTRIBUTING.md](https://github.com/google/googletest/blob/main/CONTRIBUTING.md) file which outlines the standards for source code, tests, and documentation submissions. This includes coding style, testing requirements, and commit message conventions.

4. **Submit Your Contribution**:
   - Fork the repository.
   - Develop your changes in a separate branch.
   - Include or update appropriate tests.
   - Verify your changes build successfully and pass all tests.
   - Submit a pull request with a clear description of the changes and their rationale.

5. **Engage in Review**: The maintainers will review your submission, provide feedback, and may request changes before merging.

---

## What Makes a Successful Contribution

- **Well-Scoped**: Contributions target a single, clear concern (e.g., fixing a bug or adding a specific feature). Avoid bundling unrelated changes.

- **Tests Included**: Code changes are accompanied by tests that verify the new or fixed behavior.

- **Adheres to Style and Conventions**: Follows Google's C++ style guidelines and project-specific conventions to maintain code readability and consistency.

- **Clear Documentation**: Includes comments and, if relevant, updates to usage guides or API references to help users understand the changes.

- **Thoughtful Issue Reporting**: Issue reports should be detailed with clear reproduction steps, expected behavior, and actual behavior.

- **Respectful Communication**: In discussions and code reviews, contributors maintain professional and constructive tones.

## Contributing Code

### Steps to Contribute Code

<Steps>
<Step title="Fork and Clone">
Create your own fork of the repository and clone it locally.
</Step>
<Step title="Branch Creation">
Start a new branch for your work, named descriptively for your change.
</Step>
<Step title="Make Changes">
Implement your feature, bugfix, or improvements. Write clear, maintainable code.
</Step>
<Step title="Add Tests">
Add or update tests to cover your changes and protect against regressions.
</Step>
<Step title="Build and Validate">
Build the project and run all tests to ensure no regressions.
</Step>
<Step title="Commit with Clear Messages">
Commit your changes with messages following the project's style and clearly describing what was done.
</Step>
<Step title="Push and Create Pull Request">
Push your branch to your fork and open a pull request against the main repository.
</Step>
<Step title="Participate in Review">
Respond to feedback, make requested changes promptly to help speed up merging.
</Step>
</Steps>

### Best Practices for Code Contributions

- Focus on **backward compatibility** and **non-breaking changes**.
- Write **self-explanatory code** with meaningful variable and function names.
- Use existing **utilities, macros, and idioms** familiar to GoogleTest/GoogleMock.
- Avoid unnecessary dependencies or changes unrelated to the purpose of your patch.
- Include **inline documentation** as appropriate.

## Contributing Documentation

Documentation contributions are equally vital to the project’s success and usability.

- Clearly describe new features, usage patterns, or configuration options.
- Use consistent formatting and markdown styles as seen in the documentation repository.
- Provide examples, tips, and troubleshooting advice to assist users.
- Link related topics and cross-reference API pages for ease of navigation.
- Follow the progressive disclosure model — start with simple concepts, then add detailed guidance.

---

## Reporting Issues

If you encounter bugs or have suggestions:

- Check the existing [Issue Tracker](https://github.com/google/googletest/issues) to avoid duplicates.
- Create a new issue with a descriptive title and detailed body.
- Include reproduction steps, error messages (if any), and environment details.
- Specify GoogleTest or GoogleMock version, platform, and compiler details.
- Be as concise and focused as possible.

## Practical Tips

- Always sync your fork before starting new work.
- Write tests that exercise your changes thoroughly to avoid regressions.
- Use clear, descriptive commit messages referencing relevant issues.
- Participate in reviews constructively and address feedback promptly.
- Use the project's continuous integration to verify your changes before submitting.
