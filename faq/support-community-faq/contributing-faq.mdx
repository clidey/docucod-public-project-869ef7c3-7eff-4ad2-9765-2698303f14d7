# Contributing & Reporting Issues

Contributing to the GoogleTest project means helping improve the framework by submitting fixes, enhancements, or asking questions about issues you encounter. This page guides you through the process of contributing, points to the right resources, and explains how to report bugs efficiently.

---

## How to Contribute

Contributions can range from small bug fixes and documentation updates to new features or bug reports. Before submitting your change, please ensure you follow the project guidelines to maintain quality and consistency.

### 1. Understand the Contribution Guidelines

- Always read the official [Contribution Guidelines](https://github.com/google/googletest/blob/main/CONTRIBUTING.md) (linked from the GitHub repo) before submitting any patches or pull requests.
- Follow the coding style prescribed by the project.
- Write clear, concise commit messages explaining the purpose of your changes.
- Include tests where applicable to validate your changes.

### 2. Fork, Clone, and Branch

- Fork the official repository on GitHub.
- Clone your fork to your local machine.
- Create a new branch for your work with a descriptive name.

### 3. Make Your Changes

- Implement fixes or features in your branch.
- If fixing bugs or adding features, add tests demonstrating the problem or validating the solution.
- Build and run the GoogleTest/GoogleMock tests locally to ensure nothing is broken.

### 4. Commit and Push

- Commit your changes with clear messages.
- Push your branch to your GitHub fork.

### 5. Submit a Pull Request

- Open a pull request (PR) from your branch to the main repository's `main` branch.
- Describe the change clearly in the PR description.
- Link any related issues.
- Be responsive to review comments and update your PR accordingly.

<Tip>
Good contributions include clear explanations, relevant references, and passing tests. Smaller, focused pull requests are easier to review and get merged more quickly.
</Tip>

## Reporting Bugs and Issues

Efficient issue reporting helps the maintainers understand and resolve problems faster.

### How to Report Issues Effectively

1. **Search Existing Issues**: Before opening a new issue, search the issue tracker to avoid duplicates.
2. **Provide a Clear Title**: Summarize the problem or suggestion clearly.
3. **Describe the Problem**:
   - Describe what you expected to happen.
   - Describe what actually happened.
   - Provide concrete steps to reproduce the issue including code samples if possible.
   - Include information about your environment, such as compiler version, platform, and build system.
4. **Attach Logs or Error Messages**: Include full error outputs or log snippets.
5. **Specify Versions**: Mention the version of GoogleTest and GoogleMock you are using.

### Bug Reporting Template

Here is a simple template you can use when opening an issue:

```markdown
**Issue type:** Bug / Feature Request / Question

**Description:**
A clear and concise description of the issue or suggestion.

**Steps to reproduce:**
1. Step one
2. Step two
3. ...

**Expected behavior:**
What you expected the behavior to be.

**Actual behavior:**
What happened instead.

**Environment:**
- GoogleTest version: `v1.xx.x`
- Compiler: `gcc 9.3`, `clang 12`, etc.
- Platform: Ubuntu 20.04, Windows 10, MacOS Catalina, etc.
- Build system: CMake, Bazel, Manual, other

**Additional context:**
Add any other context about the problem here.
```

<Note>
Effective bug reports speed up fixes. Avoid vague descriptions and be as detailed as possible.
</Note>

## Getting Help and Asking Questions

If you're unsure about how to use GoogleTest or have questions:

- Consult the official documentation sections such as the [GoogleTest Primer](https://google.github.io/googletest/primer.html) and [Mocking for Dummies](https://google.github.io/googletest/gmock_for_dummies.html).
- Visit community forums, GitHub Discussions, or relevant Stack Overflow tags.
- When asking questions, provide context like code snippets, error messages, and environment details.

## Maintaining Your Fork

If you contribute regularly and maintain a fork, sync with the upstream repository often to incorporate updates and avoid merge conflicts.

```bash
git remote add upstream https://github.com/google/googletest.git
git fetch upstream
git checkout main
git merge upstream/main
```
