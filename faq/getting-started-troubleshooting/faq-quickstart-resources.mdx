## FAQ: Getting Started with GoogleMock and GoogleTest

### 1. What is GoogleMock and why should I use it?
GoogleMock (gMock) is a C++ framework for creating mock classes, enabling you to simulate and verify interactions between components without depending on real objects. It helps improve test reliability by letting you precisely specify expected method calls, their arguments, and behavior.

Mocking in C++ manually can be tedious and error-prone; gMock automates this using macros and an expressive syntax inspired by frameworks like jMock for Java. Use gMock to:

- Prototype interfaces early and iterate on your design efficiently
- Test modules in isolation by mocking dependencies
- Control and verify complex interactions, including call order and argument checks
- Write flexible, fast, and maintainable tests

For a beginner-friendly introduction, see the [gMock for Dummies guide](https://google.github.io/googletest/gmock_for_dummies.html).

### 2. How do I start writing my first mock?
To use GoogleMock effectively, follow these steps:

1. **Define a Mock Class**: Use the `MOCK_METHOD` macro inside a subclass that inherits from your interface. Each virtual function you want to mock corresponds to one `MOCK_METHOD` line.
2. **Instantiate Mock Objects**: Create instances of your mock classes in your tests.
3. **Set Expectations**: Use `EXPECT_CALL` to specify which methods you expect to be called, with which arguments, how many times, and what they should do.
4. **Exercise the Code Under Test**: Run your tests so that the code calls the mock methods as expected.
5. **Automatic Verification**: GoogleMock automatically checks at the end of the test whether the expected calls happened as defined.

Example:

```cpp
#include <gmock/gmock.h>

class Turtle {
 public:
  virtual ~Turtle() {}
  virtual void PenDown() = 0;
};

class MockTurtle : public Turtle {
 public:
  MOCK_METHOD(void, PenDown, (), (override));
};

TEST(PainterTest, DrawsSomething) {
  MockTurtle turtle;
  EXPECT_CALL(turtle, PenDown()).Times(1);

  // Code under test would call turtle.PenDown()
}
```

See [gMock Cookbook - Creating Mock Classes](https://google.github.io/googletest/gmock_cook_book.html#creating-mock-classes) for detailed instructions.

### 3. What are the key macros and APIs to learn first?

- **`MOCK_METHOD`**: Defines mock methods within your mock class.
- **`EXPECT_CALL`**: Sets expectations on the mock methods, defining how they should be called and what they return/do.
  - Supports chaining clauses like `.Times()`, `.WillOnce()`, `.WillRepeatedly()`, and ordering controls.
- **`ON_CALL`**: Sets default behavior for mock methods without setting expectations.
- **Matchers**: Special objects to specify flexible argument constraints, e.g., `Eq(value)`, `_` (wildcard), or custom matchers.

Visit the [Mocking Reference](https://google.github.io/googletest/reference/mocking.html) and [gMock Cheat Sheet](https://google.github.io/googletest/gmock_cheat_sheet.html) for quick API references.

### 4. How do I deal with common issues when mocking?

- **Virtual Destructors**: Ensure interfaces you mock have virtual destructors to avoid memory leaks.
- **Order of Expectations and Calls**: Always set `EXPECT_CALL`s before exercising code; otherwise, behavior is undefined.
- **Uninteresting Calls Warning**: If mocks receive calls without expectations, you'll see warnings. Use `NiceMock<T>` wrappers to suppress these warnings when appropriate.
- **Overloaded Methods**: Use explicit argument types or type-safe matchers to disambiguate.

Consult the [Legacy gMock FAQ](https://google.github.io/googletest/gmock_faq.html) and [Troubleshooting Guide](https://google.github.io/googletest/getting_started/first-test-experience/troubleshooting-and-common-issues.html) for common pitfalls and fixes.

### 6. How do I use NiceMock, NaggyMock, and StrictMock?

These wrappers control how mock objects handle unexpected or uninteresting calls:

- **`NiceMock<T>`**: Suppresses warnings for uninteresting calls; use when you expect some calls not explicitly mentioned in expectations.
- **`NaggyMock<T>`**: (Default behavior) Warns about uninteresting calls.
- **`StrictMock<T>`**: Treats uninteresting calls as test failures; use for highly strict verification.

Example:

```cpp
using ::testing::NiceMock;
NiceMock<MockTurtle> turtle;
EXPECT_CALL(turtle, PenDown());
```

See the [Controlling Mock Call Policies](https://google.github.io/googletest/api/reference/mocking-apis/nice-strict-mocks.html) for full details.

### 7. How can I get quick help if I’m stuck?

- Enable verbose logging with the `--gmock_verbose=info` flag to get detailed call traces.
- Use your IDE or editor’s integration (e.g., Emacs setup) to jump directly to assertion failures.
- Check the output messages carefully for call order, argument mismatches, and unsatisfied expectations.
- Seek community help through official forums and GitHub discussions.

## Tips for Success

- Always define virtual destructors in your interfaces.
- Use `EXPECT_CALL` to set precise expectations; set `ON_CALL` for default behavior.
- Gradually increase test strictness: start with `NiceMock` and move to `StrictMock` as needed.
- Use matcher `_` to ignore argument values when irrelevant to your test.
- Run tests with `--gmock_verbose=info` during debugging for detailed insight.
