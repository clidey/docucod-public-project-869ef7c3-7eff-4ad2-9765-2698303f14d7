# Related Tools & Getting Involved

This page answers common user questions about ecosystem tools related to GoogleTest and GoogleMock, and provides guidance for users who want to contribute to or extend the project. It highlights useful external extensions, plug-ins, and practical advice for newcomers wishing to participate actively in the community.

## Frequently Asked Questions about Related Tools

### What related tools and extensions complement GoogleTest and GoogleMock?

GoogleTest and GoogleMock integrate with a rich ecosystem of tools that enhance your testing experience. Key ecosystem tools include:

- **Mocking Utilities**: Additional helper libraries and adapters that build on GoogleMock’s `MOCK_METHOD` macros, easing complex mock setups.
- **Matchers and Actions Extensions**: Third-party and official matchers that extend the set of built-in matchers for richer assertions.
- **Graphical Test Runners and IDE Plugins**: Plugins for popular IDEs (e.g., Visual Studio, CLion) that provide GUI-based test management, making it simpler to run, debug, and visualize test results.
- **Build System Integrations**: Tools and scripts for smooth integration with CMake and Bazel, simplifying dependency management and test execution automation.

### How do I choose between using `ON_CALL` and `EXPECT_CALL` for my mocks?

- Use `ON_CALL` to specify **default behaviors** of mock methods when you do not need to verify that the call happens.
- Use `EXPECT_CALL` to set **explicit expectations** on calls you want to verify occur during the test execution.

Remember, overusing `EXPECT_CALL` can lead to brittle tests that are hard to maintain. Favor `ON_CALL` for shared, default behavior and reserve `EXPECT_CALL` for calls critical to your test’s correctness assertion.

### What are the best practices for contributions to GoogleTest and GoogleMock?

1. **Familiarize yourself with core concepts:** Start by thoroughly understanding GoogleTest and GoogleMock’s core testing model, mocking paradigms, and assertion frameworks.
2. **Follow coding and documentation standards:** Contributions must maintain the project’s coding style, documentation quality, and comprehensive testing.
3. **Use existing issue and pull request templates:** Open issues and PRs with detailed explanations and relevant references.
4. **Engage with the community:** Participate in community forums, GitHub issues, and mailing lists to understand priority areas and ongoing development.
5. **Test your changes across supported platforms:** GoogleTest supports Linux, macOS, and Windows; assure changes compile and run successfully on these.

### How can I get help or report issues related to GoogleTest and GoogleMock?

- Check the comprehensive documentation and FAQs, especially the [Getting Started FAQ](https://github.com/google/googletest/blob/main/docs/faq/usage-and-troubleshooting/getting-started-faq.md).
- Visit the GitHub repository issues section to search for existing issues or open new ones.
- Use community platforms such as Stack Overflow with the `googletest` and `googlemock` tags.
- Join mailing lists or forums linked from the official site.

## How to Get Involved

Becoming a contributor or active participant in the GoogleTest and GoogleMock projects involves several pathways:

### Start by Using and Testing

- Use GoogleTest and GoogleMock in your projects and identify pain points or feature gaps.
- Share feedback through GitHub issues or community forums.
- Explore writing comprehensive tests to deepen your understanding.

### Contributing Code and Documentation

- Follow the [contribution guidelines](https://github.com/google/googletest/blob/main/CONTRIBUTING.md) available in the repository.
- Begin with small bug fixes, documentation improvements, or test additions.
- Submit pull requests with clear descriptions and tests covering the changes.

### Participate in Discussions

- Engage in issue discussions to offer insights, propose alternative approaches, or prioritize fixes.
- Join relevant developer mailing lists or chat groups if available.

### Share Extensions and Plugins

- Develop and share adapters, plugins, or extensions that enhance the GoogleTest / GoogleMock experience.
- Publish supporting libraries (such as matcher or action extensions) on platforms like GitHub or package managers.

## Practical Tips and Best Practices

- Leverage the `NiceMock`, `NaggyMock`, and `StrictMock` wrappers to tune mock strictness during development.
- Use the verbose flag (`--gmock_verbose`) to get detailed logs during test failures and mock interactions.
- Prefer testing interactions only when necessary, using `EXPECT_CALL` sparingly to prevent brittle tests.
- When mocking complex interfaces, consider delegating to a fake or real implementation to reduce test maintenance.
