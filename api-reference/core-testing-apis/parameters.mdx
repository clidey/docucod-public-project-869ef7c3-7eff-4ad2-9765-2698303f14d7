# Parameterized and Typed Tests

This documentation explains how to write parameterized and type-parameterized tests in GoogleTest. These advanced test forms allow you to verify your code behavior over a variety of input values and data types, ensuring comprehensive coverage with less repetitive code. You will learn how to define test cases that adapt to multiple parameters or types, instantiate them, and run them effectively.

---

## Overview

Parameterized and typed tests help you reduce boilerplate test code while improving the scope of your testing. Instead of writing multiple test cases that differ only in input values or data types, GoogleTest lets you write a single test definition that runs multiple variations automatically.

- **Parameterized Tests** focus on testing the same logic with different *input values*.
- **Typed Tests** enable testing the same logic with different *data types*.

Using these constructs, you can systematically validate your functions or classes across varied inputs or type instances, improving confidence and maintaining cleaner test suites.

---

## Writing Parameterized Tests

### Purpose

Parameterized tests allow you to run the same test logic using different sets of parameters (values), enabling data-driven testing without redundancy.

### Defining a Parameterized Test Suite

1. Derive your test fixture class from `::testing::TestWithParam<T>`, where `T` is the type of your parameter.
2. Implement your tests inside this fixture using `TEST_P` instead of `TEST_F`.
3. Use `GetParam()` inside your tests to access the current parameter.

```cpp
#include <gtest/gtest.h>

// Define a test fixture with parameter type int
class FooTest : public ::testing::TestWithParam<int> {
  // SetUp and TearDown can be declared here if needed
};

// Define parameterized test cases
TEST_P(FooTest, IsEven) {
  int n = GetParam();
  EXPECT_EQ(n % 2, 0);
}
```

### Instantiating Tests with Parameters

Use `INSTANTIATE_TEST_SUITE_P` to specify parameters for the test suite:

```cpp
INSTANTIATE_TEST_SUITE_P(
    EvenNumbersTest,    // Instance name
    FooTest,            // Test suite name
    ::testing::Values(2, 4, 6, 8));  // Parameter values
```

This will produce four test instances running the `IsEven` test with parameters 2, 4, 6, and 8.

### Supported Parameter Generators

GoogleTest provides several parameter generators:

- `Values(v1, v2, ...)` — list of values
- `Range(begin, end)` — range of values from `begin` (inclusive) to `end` (exclusive)
- `ValuesIn(container)` — values from a container or iterator range
- `Bool()` — `true` and `false`

Feel free to combine or customize generators for your test needs.

### Example: Parameterized Test for a Function

```cpp
int IsPrime(int n); // function under test

class PrimeTest : public ::testing::TestWithParam<int> {};

TEST_P(PrimeTest, ReturnsTrueForPrimes) {
  int n = GetParam();
  EXPECT_TRUE(IsPrime(n));
}

INSTANTIATE_TEST_SUITE_P(PrimeNumbers, PrimeTest, ::testing::Values(2,3,5,7,11));
```


## Writing Typed Tests

### Purpose

Typed tests enable testing templates or type-dependent code with multiple types, ensuring consistent behavior across all of them without code duplication.

### Defining Typed Test Suites

1. Use `TYPED_TEST_SUITE` to declare your typed test suite and specify the list of types.
2. Derive a test fixture template class.
3. Write typed test cases with the `TYPED_TEST` macro.

Example template fixture:

```cpp
template <typename T>
class StackTest : public ::testing::Test {
public:
  Stack<T> stack_;
};

TYPED_TEST_SUITE(StackTest, ::testing::Types<int, double, std::string>);

TYPED_TEST(StackTest, IsEmptyInitially) {
  EXPECT_TRUE(this->stack_.IsEmpty());
}

TYPED_TEST(StackTest, CanPushAndPop) {
  this->stack_.Push(TypeParam{});
  EXPECT_FALSE(this->stack_.IsEmpty());
  this->stack_.Pop();
  EXPECT_TRUE(this->stack_.IsEmpty());
}
```

### Typed Tests vs Type-Parameterized Tests

- **Typed Tests**: Use a list of types specified at suite definition.
- **Type-Parameterized Tests**: (Advanced) Let you define independent type lists and register multiple typed test suites with the same tests but different types.

### Example: Typed Test with Multiple Types

```cpp
using MyTypes = ::testing::Types<int, float, double>;
TYPED_TEST_SUITE(MyTypedTest, MyTypes);

TYPED_TEST(MyTypedTest, DefaultConstruction) {
  TypeParam value = TypeParam();
  EXPECT_EQ(value, TypeParam());
}
```

## Running Parameterized & Typed Tests

These tests are discovered and run just like normal tests with your usual test runner command (e.g. `RUN_ALL_TESTS()`). The framework will report each instantiation separately.

You can also filter tests using `--gtest_filter` with patterns matching the generated test names.

## Best Practices and Tips

- Use parameterized tests to cover multiple input values efficiently without duplicating test logic.
- Use typed tests to ensure template or type-dependent code behaves correctly for all relevant types.
- Provide meaningful instance names in `INSTANTIATE_TEST_SUITE_P` to distinguish parameter sets.
- Use generators to create dynamic parameter sets from containers or custom logic.
- Avoid over-specification in expectations within mocked methods to reduce brittle tests.
- Experiment with sequences and ordering constraints when the order of calls matters within tests.

## Troubleshooting

- Ensure parameters are copyable or default-constructible as required by the test fixture.
- When you get ambiguous overloads in typed tests, use the `Const()` wrapper or explicitly specify the overload.
- If tests are not discovered, verify correct use of `TEST_P` and `INSTANTIATE_TEST_SUITE_P` patterns.
- Use `--gtest_list_tests` to see all test instances generated.

See also: [Writing and Running Tests](/api-reference/core-testing-apis/writing-tests), [Assertions and Expectations](/api-reference/core-testing-apis/assertions), and Mocking documents for combining with mocks.
