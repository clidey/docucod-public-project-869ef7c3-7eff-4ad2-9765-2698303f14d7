# Parameterized and Typed Tests

GoogleTest provides powerful mechanisms to execute the same test logic multiple times using different inputs or types. These are essential for validating your code across a variety of conditions without duplicating test code. This page focuses exclusively on the APIs and workflows for writing **value-parameterized** and **type-parameterized** tests in GoogleTest, empowering you to create flexible, scalable, and maintainable test suites.

---

## Overview

Parameterized tests allow you to run the same test code repeatedly with different parameters. There are two main flavors:

- **Value-Parameterized Tests**: Run tests with different *values* as inputs (e.g., integers, strings, objects).
- **Type-Parameterized Tests**: Run tests for different *types* to check behavior across a set of C++ types.

Both approaches help eliminate boilerplate while expanding test coverage.

---

## Value-Parameterized Tests

### 1. Defining a Parameterized Test Fixture

Write a test fixture class template that inherits from `::testing::TestWithParam<T>`, where `T` is the parameter type:

```cpp
class FooTest : public ::testing::TestWithParam<int> {
  // Implement test setup/teardown if needed
};
```

### 2. Writing Parameterized Tests

Use the `TEST_P` macro to define tests that can access the parameter with `GetParam()`:

```cpp
TEST_P(FooTest, HandlesInputCorrectly) {
  int param = GetParam();
  EXPECT_TRUE(MyFunction(param));
}
```

### 3. Instantiating Tests with Parameters

Use `INSTANTIATE_TEST_SUITE_P` to instantiate the test suite with a name and a parameter generator:

```cpp
INSTANTIATE_TEST_SUITE_P(
  IntValues, FooTest,
  ::testing::Values(1, 2, 3));
```

The suite will run the `HandlesInputCorrectly` test 3 times with param values 1, 2, and 3.

### 4. Parameter Generators

GoogleTest provides built-in generators in `::testing` namespace to automate generating parameter sequences:

| Generator        | Description                                                    |
|------------------|----------------------------------------------------------------|
| `Range(begin, end[, step])` | Generates values from begin (inclusive) to end (exclusive) stepping by `step`. |
| `Values(v1, v2, ...)`       | Generates the specified explicit values.                     |
| `ValuesIn(container)`       | Generates values from an STL or C-style container/range.     |
| `Bool()`                   | Generates boolean values `[false, true]`.                      |
| `Combine(g1, g2, ..., gN)` | Generates Cartesian product tuples from multiple generators. |
| `ConvertGenerator<T>(g)`   | Converts generated values to type `T`, useful for custom parameter types.


### 5. Customizing Test Case Names

Enable human-readable test names by providing an optional name generator for `INSTANTIATE_TEST_SUITE_P`. The name generator must be a callable accepting a `TestParamInfo<T>` and returning a `std::string` containing only alphanumeric and underscores:

```cpp
INSTANTIATE_TEST_SUITE_P(
    NamedParams, FooTest, Values(1, 2, 3),
    [](const ::testing::TestParamInfo<int>& info) {
      return "Value_" + std::to_string(info.param);
    });
```

The generated test names will then be suffixed as e.g. `NamedParams/FooTest.HandlesInputCorrectly/Value_1`.

### 6. Example

```cpp
class IsEvenTest : public ::testing::TestWithParam<int> {};

TEST_P(IsEvenTest, ReturnsTrueForEvenNumbers) {
  int n = GetParam();
  EXPECT_EQ(n % 2, 0);
}

INSTANTIATE_TEST_SUITE_P(EvenNumbers, IsEvenTest, ::testing::Values(2, 4, 6));
```

This runs the test three times for 2, 4, and 6.

## Type-Parameterized Tests

Type-parameterized tests enable you to repeat the same test logic for multiple C++ types, allowing code validation across type variations.

### 1. Defining a Type-Parameterized Test Fixture

Define a fixture class template with a type parameter and derive from `::testing::Test`:

```cpp
template <typename T>
class NumericTest : public ::testing::Test {
 public:
  T value_{};
};
```

### 2. Associating Types With the Suite

Use `TYPED_TEST_SUITE` to associate a list of types represented by `::testing::Types<...>`:

```cpp
using NumericTypes = ::testing::Types<int, double, float>;
TYPED_TEST_SUITE(NumericTest, NumericTypes);
```

### 3. Writing Typed Tests

Use `TYPED_TEST` to define test logic. Inside test bodies, use `TypeParam` to refer to the current type:

```cpp
TYPED_TEST(NumericTest, IsDefaultZero) {
  TypeParam zero{};
  EXPECT_EQ(zero, this->value_);
}
```

### 4. Example

```cpp
template <typename T>
class ContainerTest : public ::testing::Test {};

TYPED_TEST_SUITE(ContainerTest, ::testing::Types<std::vector<int>, std::list<int>>);

TYPED_TEST(ContainerTest, InitialSizeIsZero) {
  TypeParam container;
  EXPECT_EQ(container.size(), 0);
}
```

This test runs on two container types ensuring the initial size is zero.

## Type-Parameterized Test Suites With Named Instantiations

GoogleTest supports **type-parameterized test suites** that you can define separately and instantiate multiple times with different type lists across translation units.

- Use `TYPED_TEST_SUITE_P` and `TYPED_TEST_P` to define a parameterized suite without binding types.
- Use `REGISTER_TYPED_TEST_SUITE_P` to register test names.
- Finally, instantiate with `INSTANTIATE_TYPED_TEST_SUITE_P` specifying the actual type list.

Example:

```cpp
// Define
template <typename T>
class MyTest : public ::testing::Test {};

TYPED_TEST_SUITE_P(MyTest);

TYPED_TEST_P(MyTest, DoesSomething) { /*...*/ }

REGISTER_TYPED_TEST_SUITE_P(MyTest, DoesSomething);

// Instantiate
using MyTypes = ::testing::Types<int, char>;
INSTANTIATE_TYPED_TEST_SUITE_P(MyPrefix, MyTest, MyTypes);
```

## Best Practices and Tips

- **Unique instantiation prefixes**: Use meaningful and unique prefixes in `INSTANTIATE_*_P` macros to differentiate instantiations clearly in test reports.
- **Avoid underscore in test names and parameters**: GoogleTest test and parameter names must not have underscores to avoid conflicts.
- **Use `ValuesIn` for large parameter sets**: Instead of `Values()`, prefer `ValuesIn()` when your parameters come from containers or arrays.
- **Custom parameter names improve clarity**: Provide name generators for parameterized tests to get human-readable and unique test case names.
- **Mark unused parameterized test suites**: Use `GTEST_ALLOW_UNINSTANTIATED_PARAMETERIZED_TEST` to suppress failures when a parameterized test is defined but intentionally not instantiated.

---

## Troubleshooting Common Issues

- **Tests not running for parameterized fixtures**: Ensure you have an `INSTANTIATE_TEST_SUITE_P` call with a valid parameter generator.
- **Name conflicts or collisions**: Use unique prefixes for instantiations and verify test names don’t contain invalid characters.
- **Dangling references in `ConvertGenerator` lambdas**: Avoid capturing temporaries in lambdas, specify explicit types for generated values to avoid unsafe references.
- **Test names not showing parameters**: Check if the name generator is set properly; use built-in `PrintToStringParamName` or a custom generator for better output.

Parameterized and type-parameterized tests unlock test code reuse, making it trivial to run the same scenario across diverse inputs and types. By mastering these APIs, your test suite can grow naturally without redundancy while improving coverage and maintainability.
