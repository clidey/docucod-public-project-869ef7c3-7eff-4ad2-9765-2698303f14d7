# Using Parameterized and Type-Parameterized Tests

Unlock efficient test coverage with parameterized and type-parameterized tests in GoogleTest. This guide walks you through real scenarios, setup, and maintenance of tests that run with multiple values or types, improving code robustness and reducing duplication.

---

## 1. Understanding Parameterized Tests

### What They Are
Parameterized tests let you write a single test logic that runs multiple times with different input values. Instead of duplicating the same test with different data, parameterized tests improve maintainability and test coverage.

### When to Use
- You want to verify the same logic against a variety of input values.
- Avoid duplicated test code for similar behavior with multiple inputs.
- Ensure consistent testing for edge cases, typical cases, and invalid inputs.

### Expected Outcome
By implementing parameterized tests, your test suite becomes more comprehensive and easier to update. It reduces the chance of missing test scenarios as coverage scales.

---

## 2. Setting Up Parameterized Tests

### Prerequisites
- Your project is already configured to use GoogleTest.
- Basic familiarity with GoogleTest's test macros.

### Implementation Steps

<Steps>
<Step title="Define the Parameterized Test Fixture Class">
Create a test fixture inheriting from `::testing::TestWithParam<T>` where `T` is the type of the parameter.

```cpp
class MyParameterizedTest : public ::testing::TestWithParam<int> {
  // Optional: Declare setup/teardown if needed.
};
```

This class will be the basis for multiple test runs with different parameter values.
</Step>

<Step title="Write Tests Using the Parameter">
Use the `GetParam()` method within the test body to access the current parameter value.

```cpp
TEST_P(MyParameterizedTest, IsPositive) {
  int value = GetParam();
  EXPECT_GT(value, 0);
}
```
</Step>

<Step title="Register Test Values">
Register the parameter values you want to test with `INSTANTIATE_TEST_SUITE_P`.
Example with integer values:

```cpp
INSTANTIATE_TEST_SUITE_P(
    PositiveValues,       // Instance name
    MyParameterizedTest,  // Test case name
    ::testing::Values(1, 5, 10, 100));
```

This causes GoogleTest to run `IsPositive` test for each entry.
</Step>

<Step title="Build and Run Tests">
Compile your tests. GoogleTest will show each parameterized test execution separately with the parameter listed.

```bash
./my_test_binary --gtest_filter=MyParameterizedTest.*
```
</Step>
</Steps>

### Verification
- Each parameter causes an independent test run.
- Failures show the parameter value causing failure.

## 3. Type-Parameterized Tests

### Purpose
Sometimes you want to test the same logic across different types (e.g., templates). Type-parameterized tests let you write tests once and run them for multiple types.

### Setup Steps

<Steps>
<Step title="Define a Test Template Class">
Create a template test class inheriting from `::testing::Test`:

```cpp
template <typename T>
class MyTypedTest : public ::testing::Test {
  // Implementation can use T
};
```
</Step>

<Step title="Define the Type List">
Create a `typedef` or `using` alias with the list of types you want to test:

```cpp
using MyTypes = ::testing::Types<int, float, double>;
```
</Step>

<Step title="Register the Typed Test Suite">
Register the typed test suite and associate it with the type list:

```cpp
TYPED_TEST_SUITE(MyTypedTest, MyTypes);
```
</Step>

<Step title="Write Typed Tests Using TYPED_TEST">
Write typed tests using `TYPED_TEST`. Use `TypeParam` to refer to the current type.

```cpp
TYPED_TEST(MyTypedTest, IsDefaultConstructible) {
  TypeParam var{};  // test default construction
  EXPECT_TRUE(true);  // example assertion
}
```
</Step>
</Steps>

### Result
GoogleTest will instantiate `MyTypedTest` for each listed type in `MyTypes`.

## 5. Best Practices and Tips

- **Keep Parameters Focused:** Only parameterize what's necessary to avoid overly complex tests.
- **Name Clearly:** Use descriptive instance names in `INSTANTIATE_TEST_SUITE_P` to clarify test intent.
- **Use Parameter Generators:** Besides `Values()`, use generators like `Range()`, `ValuesIn()`, and `Bool()` for flexible parameter spaces.
- **Retire Expectations Appropriately:** When mixing mocks, ensure expectations don't persist beyond intended calls to avoid flaky results.
- **Avoid Over-Specification:** Parameterized tests should not verify too many things at once to keep failures easy to diagnose.

## 6. Troubleshooting

<AccordionGroup title="Common Issues and Solutions">
<Accordion title="Parameterized Test Not Running Any Cases">
Make sure you correctly register parameter values with `INSTANTIATE_TEST_SUITE_P`. Forgot this, no tests will be executed.
</Accordion>
<Accordion title="Test Fails Without Clear Parameter Info">
Ensure your test names and parameter instances are descriptive. Use pretty printers or custom parameter naming if applicable.
</Accordion>
<Accordion title="Type Parameterized Tests Compilation Errors">
Verify all template types satisfy test fixture requirements and that the test methods are valid for all types.
</Accordion>
<Accordion title="Unexpected Mock Failures in Parameterized Tests">
Check that mock expectations are appropriately scoped per parameter run and that mocks reset state between tests.
</Accordion>
</AccordionGroup>

## 7. Examples

### Parameterized Test Example

```cpp
#include <gtest/gtest.h>

class IsEvenTest : public ::testing::TestWithParam<int> {};

TEST_P(IsEvenTest, ReturnsTrueForEvenNumbers) {
  int n = GetParam();
  EXPECT_EQ(n % 2, 0);
}

INSTANTIATE_TEST_SUITE_P(
    EvenNumbers,
    IsEvenTest,
    ::testing::Values(2, 4, 6, 8, 10));
```

### Type-Parameterized Test Example

```cpp
#include <gtest/gtest.h>

template <typename T>
class NumericLimitsTest : public ::testing::Test {};

using MyTypes = ::testing::Types<int, float, double>;
TYPED_TEST_SUITE(NumericLimitsTest, MyTypes);

TYPED_TEST(NumericLimitsTest, MinimumIsLessThanMaximum) {
  EXPECT_LT(std::numeric_limits<TypeParam>::min(),
            std::numeric_limits<TypeParam>::max());
}
```
