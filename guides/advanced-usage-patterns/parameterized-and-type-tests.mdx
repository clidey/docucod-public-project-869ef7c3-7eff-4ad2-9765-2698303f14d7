# Parameterized and Type-Parameterized Tests

Master running your tests across multiple data values and types. This guide shows how to create parameterized test suites and apply them for broader code coverage, improving test reliability with less repetition.

---

## Overview

GoogleTest offers powerful features to run the same tests multiple times with different inputs or types. This helps increase your code coverage and find issues that only manifest under certain data conditions or type assumptions.

This page focuses only on **parameterized tests** (which vary test data) and **type-parameterized tests** (which vary test types), explaining how to write them, instantiate them with test data or types, and customize their behavior and naming.


## Prerequisites

- Basic familiarity with writing GoogleTest tests, including TEST(), TEST_F() macros, and test fixtures.
- GoogleTest installed and configured in your project.
- Understanding of how standard tests work (see [First Unit Test guide](../getting-started/configuration-initial-run/first-test)).


## Expected Outcome

By the end of this guide, you will be able to:

- Write tests that automatically run multiple times with different input values.
- Write tests that automatically run against multiple C++ types.
- Customize how test names are generated for parameterized instances.
- Use GoogleTest’s built-in parameter generators as well as combine multiple parameter sets.


## Time Estimate

15-30 minutes to create your first parameterized test and understand the basics of type-parameterized tests.

## Parameterized Tests

Parameterized tests let you run the same logic multiple times with different values. Instead of copying the test code, you define the inputs once and GoogleTest expands the tests automatically.

### Step 1: Define a Parameterized Test Fixture

Create a test fixture inheriting from `::testing::TestWithParam<T>` where `T` is your parameter type.

```cpp
class FooTest : public ::testing::TestWithParam<int> {
  // Optional setup/teardown and members here
};
```

### Step 2: Write Your Parameterized Tests

Use the `TEST_P` macro instead of `TEST` or `TEST_F`. Inside, access the current parameter with `GetParam()`.

```cpp
TEST_P(FooTest, HandlesPositiveNumbers) {
  int value = GetParam();
  EXPECT_GT(value, 0);
  EXPECT_EQ(value % 2, 0) << "Expect even number";
}
```

You can write multiple `TEST_P` tests under the same fixture.

### Step 3: Instantiate Your Parameterized Test Suite

Use `INSTANTIATE_TEST_SUITE_P` to bind a set of parameters to your test suite.

```cpp
INSTANTIATE_TEST_SUITE_P(EvenValues, FooTest, ::testing::Values(2, 4, 6, 8));
```

This runs all your `TEST_P` tests in `FooTest` four times with values 2, 4, 6, and 8.


### Parameter Generators

GoogleTest provides several built-in generators to define parameter sets:

| Generator            | Description                                   | Example                       |
| -------------------- | ---------------------------------------------|----------------------------- |
| `Range(begin, end, step)` | Produces values from `begin` to before `end` with step size `step` (default 1). Does not include `end`.
| `Values(v1, v2, ..., vN)` | Produces an explicit list of values.
| `ValuesIn(container)` or `ValuesIn(begin, end)` | Produces values from an array, STL container, or iterator range.
| `Bool()`              | Produces `false` and `true`.
| `Combine(g1, g2, ..., gN)` | Produces the Cartesian product of multiple generators combined as tuples.
| `ConvertGenerator<T>(gen)` or `ConvertGenerator(gen, func)` | Converts the generated values to desired type `T`, optionally using a conversion function.


### Example: Using Multiple Generators and Tuple Parameters

```cpp
class AnimalTest : public ::testing::TestWithParam<std::tuple<const char*, int>> {};

TEST_P(AnimalTest, CheckAge) {
  const char* name;
  int age;
  std::tie(name, age) = GetParam();
  EXPECT_GE(age, 0);
}

INSTANTIATE_TEST_SUITE_P(
  AnimalVariations, AnimalTest,
  ::testing::Combine(::testing::Values("cat", "dog"), ::testing::Values(1, 2)));
```

This runs tests for all combinations: ('cat',1), ('cat',2), ('dog',1), ('dog',2).


### Step 4: (Optional) Customize Test Naming

You can provide a name generator function or functor that produces readable test names based on parameters.

```cpp
INSTANTIATE_TEST_SUITE_P(
  CustomNames, FooTest, ::testing::Values(1, 2),
  [](const ::testing::TestParamInfo<int>& info) {
    return "Val" + std::to_string(info.param);
  });
```

Note: Names must consist only of alphanumeric characters and underscores.

## Type-Parameterized Tests

These tests allow you to write a test suite that runs once for each type in a list, useful for testing templates or concepts.


### Step 1: Define a Fixture Template

Create a class template deriving from `::testing::Test`.

```cpp
template <typename T>
class NumericTest : public ::testing::Test {
 public:
  T value_ = T();
};
```


### Step 2: Associate Types to Your Test Suite

Define a type list and attach it using `TYPED_TEST_SUITE`.

```cpp
using NumericTypes = ::testing::Types<int, double, long>;
TYPED_TEST_SUITE(NumericTest, NumericTypes);
```


### Step 3: Write Typed Tests

Use `TYPED_TEST` to define tests. Inside tests, use `TypeParam` to refer to the current type.

```cpp
TYPED_TEST(NumericTest, DefaultValueIsZero) {
  EXPECT_EQ(TypeParam(), this->value_);
}
```

You can define multiple tests for the fixture template.


### Step 4: Run Tests

Tests automatically run once for each type in your list.


### Type-Parameterized Test Patterns

Alternatively, you can define abstract type-parameterized test suites with `TYPED_TEST_SUITE_P` and `TYPED_TEST_P` if you want to register test patterns separately and instantiate them later:

```cpp
template <typename T>
class FooTest : public ::testing::Test {};

TYPED_TEST_SUITE_P(FooTest);

TYPED_TEST_P(FooTest, DoesBlah) { /* test code using TypeParam */ }

REGISTER_TYPED_TEST_SUITE_P(FooTest, DoesBlah);

using MyTypes = ::testing::Types<int, char>;
INSTANTIATE_TYPED_TEST_SUITE_P(MyInstantiation, FooTest, MyTypes);
```

This is useful for generic interfaces that different implementations instantiate.


### Optional: Custom Type Name Generation

You can provide a class with a `static std::string GetName(int index)` template to customize type names.

```cpp
class TypeNameGenerator {
 public:
  template <typename T>
  static std::string GetName(int) {
    if (std::is_same<T, int>::value) return "int";
    if (std::is_same<T, char>::value) return "char";
    return "unknown";
  }
};

TYPED_TEST_SUITE(NumericTest, NumericTypes, TypeNameGenerator);
```

## Troubleshooting & Tips

<AccordionGroup title="Common Issues and Solutions">
<Accordion title="Tests Not Running as Expected in Parameterized Tests">
Ensure that your parameterized test suite is instantiated with `INSTANTIATE_TEST_SUITE_P`. Without this, tests won’t be generated.
</Accordion>
<Accordion title="Compiler Errors with Type-Parameterized Tests">
Verify you included the correct macros: `TYPED_TEST_SUITE` before test definitions, `TYPED_TEST` for test cases, and `INSTANTIATE_TYPED_TEST_SUITE_P` for instantiation.
</Accordion>
<Accordion title="Parameter Generator Produces No Tests">
Check that your parameter generator actually produces values (e.g., non-empty vector or meaningful Range). Empty sets produce no tests.
</Accordion>
</AccordionGroup>

<Tip>
Use `Combine()` to produce all combinations of multiple parameter sets and achieve comprehensive testing with minimal code.
</Tip>

<Tip>
Prefer `TEST_P` and parameterized tests over writing repetitive tests with varied inputs to reduce maintenance and improve clarity.
</Tip>

<Note>
`GetParam()` is only valid inside `TEST_P` test bodies and returns a const reference to the current parameter.
</Note>

<Warning>
Custom parameterized test names must use only alphanumeric characters and underscores to avoid test registration errors.
</Warning>
