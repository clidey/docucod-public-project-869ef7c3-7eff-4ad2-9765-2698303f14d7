# Running Tests: Command Line & Output

This guide helps you run your GoogleTest-based tests from the command line, understand the test runner output, and customize test execution to fit your workflow. Whether you're launching a single test or managing complex test suites, these actionable instructions demystify the process and empower you to interpret test results effectively.

---

## 1. Running Tests from the Command Line

GoogleTest tests are typically compiled into an executable that runs your tests. You invoke this executable in your terminal or command prompt.

### Step-by-Step Instructions

1. **Build Your Test Executable**
   - Compile and link your test code using your build system (e.g., CMake, Bazel) to produce the test executable.

2. **Run All Tests**
   - Simply run the executable without any arguments:

   ```bash
   ./your_test_executable
   ```

   - This command runs all registered tests and prints detailed results.

3. **Run Specific Tests by Name**
   - Use the `--gtest_filter` flag to run tests matching a pattern:

   ```bash
   ./your_test_executable --gtest_filter=TestSuiteName.TestName
   ```

   - To run multiple tests, separate patterns with `:` (colon):

   ```bash
   ./your_test_executable --gtest_filter=Suite1.*:Suite2.Test1
   ```

   - Negative patterns are supported when prefixed with `-`:

   ```bash
   ./your_test_executable --gtest_filter=-Suite3.*
   ```

4. **List All Available Tests**
   - Use the `--gtest_list_tests` flag:

   ```bash
   ./your_test_executable --gtest_list_tests
   ```

   - This outputs all test suites and tests registered in the executable.

5. **Control Test Output Verbosity**
   - Use `--gtest_verbose` or the Google Mock-specific `--gmock_verbose` flag to adjust log detail (more on this below).

### Troubleshooting

- **Test executable does not run or crashes:**
  - Confirm the build completed successfully.
  - Ensure all dependencies and mock classes are linked.
  - Verify your environment matches the prerequisites of your code.

- **No tests run or tests not discovered:**
  - Check that your tests are properly registered using `TEST()` or `TEST_F()` macros.
  - Ensure you did not filter out all tests inadvertently with `--gtest_filter`.

## 2. Interpreting Test Output

GoogleTest outputs structured logs that help you determine the health and behavior of your tests.

### Understanding Key Output Elements

- **[ RUN ]**: Indicates a test case/test suite is starting.

- **[       OK ]**: The test passed without verification or execution errors.

- **[  FAILED  ]**: The test encountered failed assertions or errors.

- **Summary Lines**:
  - Shows the number of tests run, passed, failed, and the total time taken.

### Sample Output

```shell
[ RUN      ] FactorialTest.HandlesZeroInput
[       OK ] FactorialTest.HandlesZeroInput (0 ms)
[ RUN      ] FactorialTest.HandlesPositiveInput
[  FAILED  ] FactorialTest.HandlesPositiveInput (1 ms)
...
[==========] 5 tests from 1 test suite ran. (5 ms total)
[  PASSED  ] 4 tests.
[  FAILED  ] 1 test, listed below:
[  FAILED  ] FactorialTest.HandlesPositiveInput
``` 

### Detailed Failure Information

- When a test fails, GoogleTest prints the file name and line number.
- It also shows the values involved and a message explaining the failure.
- For verbose output, use `--gtest_brief` or `--gtest_verbose=info`.

### Customizing Output Format

- **Quiet Mode:**
  - Use `--gtest_output=xml[:PATH]` to generate XML reports for CI pipelines or coverage tools:

  ```bash
  ./your_test_executable --gtest_output=xml:report.xml
  ```

- **Color Options:**
  - Force color output:

  ```bash
  ./your_test_executable --gtest_color=yes
  ```

  - Disable colors:

  ```bash
  ./your_test_executable --gtest_color=no
  ```

## 3. Common Command Line Flags to Streamline Testing

### Test Selection

| Flag              | Purpose                                      | Example Usage                            |
|-------------------|----------------------------------------------|----------------------------------------|
| `--gtest_filter=`  | Run only tests matching the pattern          | `--gtest_filter=MySuite.Test1`         |
| `--gtest_list_tests` | List all available tests                      | `--gtest_list_tests`                    |

### Output Control

| Flag                  | Purpose                             | Example Usage                    |
|-----------------------|-------------------------------------|---------------------------------|
| `--gtest_output=`     | Generates output reports (XML/JSON) | `--gtest_output=xml:report.xml` |
| `--gtest_color=`      | Set console color output             | `--gtest_color=yes`              |

### Verbosity & Logging

GoogleMock offers verbosity control for mock object logging:

| Flag                  | Effect                                           | Example Usage                  |
|-----------------------|-------------------------------------------------|-------------------------------|
| `--gmock_verbose=info`    | Prints info, warnings, and errors. Shows stack traces for uninteresting calls. | `--gmock_verbose=info`          |
| `--gmock_verbose=warning` | Prints warnings and errors, but hides stack traces (default).          | `--gmock_verbose=warning`       |
| `--gmock_verbose=error`   | Prints errors only. Less verbose.                              | `--gmock_verbose=error`         |

Set these also programmatically in code:

```cpp
::testing::FLAGS_gmock_verbose = "error";
```

### Other Useful Flags

- `--gtest_repeat=N` runs tests N times (useful for detecting flaky tests).
- `--gtest_shuffle` randomizes test execution order, helping expose order dependencies.
- `--gtest_break_on_failure` stops execution on the first failed test.

## 4. Advanced Usage: Customizing Test Behavior

### Running Subsets of Tests
Use wildcards for flexible filtering:

```bash
./your_test_executable --gtest_filter=SuiteName.*
./your_test_executable --gtest_filter=SuiteName.TestA:SuiteName.TestB
./your_test_executable --gtest_filter=-SuiteName.UnwantedTest*
```

### Running Tests with Different Output Formats

You can generate XML reports for integration with CI/CD systems and dashboards:

```bash
./your_test_executable --gtest_output=xml
./your_test_executable --gtest_output=json:custom_report.json
```

### Controlling Mock Call Logging

Refine how much GoogleMock logs about mock method calls with the `--gmock_verbose` flag:

- Use `info` for verbose messages during mock expectations and calls.
- Use `warning` (default) to show only warnings and errors.
- Use `error` to minimize output to errors only.

This helps balance noise and information based on your debugging needs.

### Integration with Development Tools

- Many IDEs can integrate GoogleTest executables and parse the output.
- XML output can be imported into CI tools such as Jenkins or GitHub Actions.

## 5. Tips & Best Practices

- **Always explicitly initialize GoogleTest/GoogleMock:**

  ```cpp
  int main(int argc, char** argv) {
    ::testing::InitGoogleMock(&argc, argv);
    return RUN_ALL_TESTS();
  }
  ```

- **Avoid running tests with partially constructed mocks:** Failures may be silent or misleading.
- **Use `--gtest_list_tests` to discover test names when filtering.**
- **Be mindful** that filtering does not compile out tests; all test code is still linked.
- **Check output verbosity** if you are not seeing expected mock call details.
- **Use `--gtest_repeat` and `--gtest_shuffle` to detect flaky tests.**

## 6. Troubleshooting Common Issues

### Tests Do Not Run
- Verify tests are registered using GoogleTest macros.
- Check that you do not accidentally filter out all tests.
- Confirm GoogleTest initialization is called in `main()`.

### Unexpected Output or Missing Details
- Adjust verbosity flags `--gmock_verbose`.
- Enable color to improve readability (`--gtest_color=yes`).

### Test Executable Crashes or Hangs
- Run tests individually using `--gtest_filter`.
- Confirm your mocks and test fixtures are correctly constructed.

### Test Results Are Confusing
- Use `--gtest_catch_exceptions=1` to catch unexpected exceptions.
- Utilize `--gtest_break_on_failure` to stop immediately upon failure.
