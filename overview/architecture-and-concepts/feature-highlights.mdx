# Feature Highlights

GoogleTest is a powerful and flexible C++ testing framework designed to streamline the development and maintenance of high-quality software. This page highlights the core features that make GoogleTest an essential tool for C++ developers aiming to write robust, maintainable, and verifiable code.

---

## xUnit-style Structure and Test Discovery

GoogleTest structures tests following the widely-adopted xUnit framework principles â€” tests are organized into test cases and test suites which helps you logically group related tests and share setup/teardown code via fixtures.

- **Automatic test discovery** enables effortless execution of tests without manually listing them.
- Seamless integration with IDEs and continuous integration pipelines accelerates development cycles.

Example snippet to define a simple test:

```cpp
#include <gtest/gtest.h>

TEST(SampleTestSuite, SampleTestCase) {
  EXPECT_EQ(4, 2 + 2);
}
```

## Rich Assertions and User-Defined Assertions

GoogleTest offers an extensive suite of assertions that verify conditions and provide detailed diagnostic messages. Common assertions include `EXPECT_EQ`, `ASSERT_TRUE`, and `EXPECT_THROW`, each tailored for different test conditions.

You can create **custom assertions** to express domain-specific validation, enhancing test clarity and maintainability.

Example of a simple assertion:

```cpp
EXPECT_NEAR(actual_value, expected_value, tolerance);
```

## Death Tests

GoogleTest supports death tests, which verify that your program behaves correctly under fatal error conditions, such as invalid input or failed assertions. This capability enables thorough validation of error handling paths without crashing the test suite.

Example:

```cpp
EXPECT_DEATH({ SomeDangerousOperation(); }, "expected error message");
```

## Parameterized and Type-Parameterized Tests

To reduce boilerplate and increase coverage, GoogleTest supports:

- **Value-Parameterized Tests:** Run the same test logic with different input values.
- **Type-Parameterized Tests:** Test templated code by running tests with different types.

This promotes DRY principles and encourages exploratory testing through simple constructs.

Example of a parameterized test:

```cpp
class IsEvenTest : public ::testing::TestWithParam<int> {};

TEST_P(IsEvenTest, ReturnsTrueForEvenNumbers) {
  EXPECT_EQ(0, GetParam() % 2);
}

INSTANTIATE_TEST_SUITE_P(EvenNumbers, IsEvenTest, ::testing::Values(2, 4, 6, 8));
```

## Mock Support Using GoogleMock

GoogleTest includes GoogleMock, a feature-rich mocking framework that enables:

- **Declarative mock class and method definitions** using straightforward macros.
- **Precise control over mock behavior and expectations**, including call order and frequency.
- Support for complex argument matchers, actions, and sequences.
- Strictness models for controlling test sensitivity to uninteresting or unexpected calls (`NiceMock`, `NaggyMock`, `StrictMock`).

Example of a mock class and setting expectations:

```cpp
#include <gmock/gmock.h>

class MockTurtle {
 public:
  MOCK_METHOD(void, PenDown, (), ());
  MOCK_METHOD(void, Forward, (int distance), ());
};

TEST(TurtleTest, MovesForward) {
  MockTurtle turtle;
  EXPECT_CALL(turtle, PenDown());
  EXPECT_CALL(turtle, Forward(100));

  Painter painter(&turtle);
  painter.DrawLine();
}
```

## Strictness Models: Nice, Naggy, and Strict Mocks

GoogleMock allows you to tune mock behavior when unexpected or uninteresting calls occur:

- **NiceMock:** Suppresses warnings for uninteresting calls, enabling less noisy tests.
- **NaggyMock:** (default) Warns on uninteresting calls but does not fail the test.
- **StrictMock:** Treats uninteresting calls as errors, enforcing rigorous call verification.

These modes help tailor tests from lenient during development to strict for final validation.

## Flexible Test Runners and Verbosity Controls

GTest and GoogleMock offer configurable test runners, supporting:

- Running all or selected tests via command line.
- Output verbosity levels to control the detail of test results and mock call traces.
- Integration with IDEs, scripts, and CI workflows.

Example command to run tests with verbose output:

```shell
test_binary --gtest_filter=MyTestSuite.* --gmock_verbose=info
```
